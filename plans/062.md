## Root Cause Analysis

When barf runs `auto` on the cyrus project, the verification commands (`cyrus2/.venv/bin/python -m ruff ...`) fail because `projectCwd` is dropped at four points:

1. **`barf-worker.ts` line 217**: `auto` case passes `deps: { db }` — omits `projectCwd: projectDir`
2. **`auto.ts` line 76-82**: `AutoDeps` type doesn't include `projectCwd` field
3. **`auto.ts` line 125-135**: `loopDeps` construction omits `projectCwd`
4. **`auto.ts` line 435-439**: Direct `_verify()` call for BUILT issues omits `deps: { cwd }`
5. **`pre-complete.ts` line 58**: `execFn('sh', ['-c', step.command])` omits `cwd` third arg

Without `cwd`, the spawned process inherits barf server's working directory. Relative paths like `cyrus2/.venv/bin/python` cannot resolve → `sh: 1: cyrus2/.venv/bin/python: not found`.

## Approach

Thread `projectCwd` through the auto command chain, mirroring the pattern already used by the `plan`/`build`/`verify` worker commands. Also fix the fix-command execution in `pre-complete.ts`. This is a minimal, surgical fix — four source files, each change is 1-3 lines.

**Why this approach over alternatives**:
- Alternative: Change `.barfrc` to use absolute paths → Only fixes cyrus, doesn't fix the systemic bug
- Alternative: Change `process.cwd()` before spawning → Global mutation, affects all concurrent workers
- Chosen: Thread `projectCwd` → Matches existing DI pattern (`.claude/rules/dependency-injection.md`), fixes all projects, no global state

      callCount++
      if (callCount <= 2) {

      return Promise.resolve([])
    },
  })

    provider,
    batch: 1,
    auditGate: false,
    config: defaultConfig(),
    deps: { ...deps, verifyIssue: mockVerify as never, projectCwd: '/my/project' },
  })

  expect(capturedOpts).not.toBeNull()
