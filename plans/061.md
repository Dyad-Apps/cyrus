## Gap Analysis

**Already exists**:
- `projectCwd` is properly threaded through the `runLoop` → `handleBuildCompletion` → `verifyIssue` path when called from the **worker's direct build/plan/verify commands** (barf-worker.ts lines 99-209)
- `preCheckCommand` correctly resolves relative paths when `cwd` is provided
- `verifyIssue` accepts `deps.cwd` and passes it through to `runVerification`
- Tests exist for `preCheckCommand`, `runVerification`, `handleBuildCompletion`, and `autoCommand` — but none test that `projectCwd` is threaded correctly through auto

**Needs building**:
1. `autoCommand` does NOT receive or thread `projectCwd` — this is the root cause of the verification failure
2. The auto command's verify loop calls `verifyIssue` without `deps` (no `cwd`)
3. `loopDeps` built inside `autoCommand` omits `projectCwd`, so `runLoop` → `handleBuildCompletion` also lacks it
4. The worker dispatches `autoCommand` without passing `projectDir`
5. `verifyWithSession` (exported, called by auto refactors) doesn't accept or pass `cwd`
6. **Cyrus formatting**: 3 files fail `ruff format --check` — `cyrus_config.py`, `cyrus_server.py`, `test_027_cyrus_config.py` (pre-existing cosmetic issues blocking CHECK_COMMAND)

## Root Cause

The error `sh: 1: cyrus2/.venv/bin/python: not found` occurs because `autoCommand` in `auto.ts` never receives or threads `projectCwd`. Three specific code paths fail:

1. **Auto verify loop** (auto.ts:434-439): `_verify({ issueId, config, provider })` — no `deps`, no `cwd`
2. **loopDeps** (auto.ts:125-135): Built without `projectCwd`, so `runLoop` for build/plan from auto has `deps.projectCwd = undefined`
3. **Worker dispatch** (barf-worker.ts:211-219): `autoCommand({ ..., deps: { db } })` — `projectDir` not passed

When `cwd` is undefined, `sh -c "cyrus2/.venv/bin/python ..."` runs relative to the barf process's working directory instead of the project directory, so the relative path fails.

## Approach

Thread `projectCwd` from the worker through `autoCommand` to all verification and build paths. This is a straightforward plumbing fix with no architectural changes — the pattern already exists in the direct build/verify worker commands.

**Why this approach**: The existing `RunLoopDeps.projectCwd` pattern is proven and well-documented. We just need to apply it consistently to the `autoCommand` path. No new abstractions needed.
