**Broken window** (same code path, small fix):

6. **`verifyWithSession` doesn't thread cwd** (`orchestration.ts:433`) — calls `verifyIssue({ issueId, config, provider })` without any `deps`; latent bug for gate-phase verify calls

## Root Cause

When barf's auto loop runs verification on BUILT issues, `cwd` is never passed to `verifyIssue()`. The `preCheckCommand()` function receives `cwd=undefined`, so relative paths in custom `.barfrc` commands (like `cyrus2/.venv/bin/python`) are not resolved against the project directory. The error message confirms this: "resolved to cyrus2/.venv/bin/python" (same as input — no `cwd` prefix applied).

## Approach

**Thread `projectCwd` through the `auto` command path.** The fix follows the exact same pattern already used by the `plan`/`build`/`verify` worker cases — we're patching a gap where `auto` was overlooked.

**Design decision**: Add `projectCwd` to `AutoCommandOpts` (not `AutoDeps`) because per the DI naming convention (`dependency-injection.md`), `*Deps` = injectable behavior (functions, services), `*Opts` = configuration (settings, flags, IDs). `projectCwd` is a configuration value, not injectable behavior.

**Why not fix the cyrus `.barfrc` instead?** The `.barfrc` is correct — relative paths are valid and work when `cwd` is set properly. The bug is in barf-ts's plumbing, not the cyrus config.
