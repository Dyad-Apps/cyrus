## Root Cause

The error `Command not found: cyrus2/.venv/bin/python (resolved to cyrus2/.venv/bin/python)` occurs because the auto command path never threads `projectCwd` to verification. The "resolved to" value being identical to the raw token proves `cwd` is `undefined` (no prefix applied).

**Broken call chain** (auto command → verify):
```
barf-worker.ts:211  → autoCommand({ ..., deps: { db } })           -- no projectDir
auto.ts:90          → AutoCommandOpts has no projectCwd field       -- can't receive it
auto.ts:125         → loopDeps built without projectCwd             -- runLoop builds also broken
auto.ts:435         → _verify({ issueId, config, provider })       -- no deps.cwd
orchestration.ts:64 → cwd = deps?.cwd → undefined                  -- falls through
checks.ts:98        → resolvedPath = firstToken (no cwd prefix)    -- FAILS
```

**Working comparison** (direct verify command):
```
barf-worker.ts:192  → verifyIssue({ ..., deps: { cwd: projectDir } })  -- projectDir passed
orchestration.ts:64 → cwd = deps?.cwd → "/home/.../cyrus"              -- resolved
checks.ts:98        → resolvedPath = "/home/.../cyrus/cyrus2/.venv/..." -- WORKS
```

## Approach

Surgical plumbing fix: thread `projectCwd` through the auto command path, replicating the pattern already established by the direct worker commands. No architectural changes, no new abstractions.

**Why this approach**: The `RunLoopDeps.projectCwd` and `VerifyOpts.cwd` patterns are proven and well-documented. The fix is to apply them consistently to the `autoCommand` path which was missed during the auto-command simplification (plan 172).
