# Implementation Plan: Fix verification failures — CWD not threaded to verification commands

**Issue**: [058](/home/daniel/Projects/barf/cyrus/issues/058.md)
**Created**: 2026-03-17
**PROMPT**: PROMPT_plan

## Gap Analysis

**Already exists**:
- `execFileNoThrow()` correctly handles optional `cwd` parameter (passes it to `spawn()`)
- `runVerification()` in `checks.ts` accepts and passes `cwd` to `execFn`
- `runPreComplete()` accepts `cwd` in opts and passes it to `runVerification()`
- `verifyIssue()` accepts `deps?: VerifyOpts` with `cwd` and threads it through `makeVerifyPhases()` → `runChecksPhase()` → `runPreComplete()`
- `RunLoopDeps` has `projectCwd?: string` — used by `runLoop()` call sites (`outcomes.ts`, `loop.ts`)
- `barf-worker.ts` extracts `projectDir` from `WorkerRequest` and passes `cwd: projectDir` for the `verify` case
- Tests exist for `runVerification` CWD forwarding and `verifyIssue` CWD forwarding

**Needs building** (4 bugs, all in barf-ts):

1. **`pre-complete.ts` line 58** — Fix commands (`FIX_COMMANDS`) don't receive `cwd` even when it's available. The `cwd` is destructured on line 52 but only passed to `runVerification()` (line 78), not to fix command execution (line 58).

2. **`orchestration.ts` line 433** — `verifyWithSession()` calls `verifyIssue({ issueId, config, provider })` WITHOUT `deps`. The `VerifyWithSessionOpts` interface also lacks a `deps` field, so there's no way to pass CWD through this wrapper.

3. **`auto.ts` lines 125-135, 435-439** — `autoCommand()` has two CWD gaps:
   - `loopDeps` construction omits `projectCwd` (all `runLoop()` calls from auto get no CWD)
   - Inline verify call at line 435 passes no `deps` (CWD never reaches verification)
   - `AutoDeps` type lacks a `projectCwd` field

4. **`barf-worker.ts` line 211-218** — Auto case dispatches `autoCommand({ ..., deps: { db } })` without `projectCwd`, despite having `projectDir` available from the `WorkerRequest`.

**Impact**: Any project using relative paths in `.barfrc` commands (like `cyrus2/.venv/bin/python`) fails all verification when triggered via the `auto` command path. The `runLoop` path works (via `outcomes.ts` which passes CWD) but `auto`'s standalone verify for BUILT issues and all fix commands are broken.

## Approach

Thread `projectCwd` through the entire auto command path, matching the pattern already established in `runLoop`/`outcomes.ts`. Also fix the pre-complete fix command CWD bug which affects all code paths.

**Why this approach**: The `projectCwd` pattern already exists and works in `runLoop`/`outcomes.ts`. The auto command simply failed to adopt it. This is a plumbing fix, not an architectural change.

## Key Decisions

- **Selected approach**: Thread existing `projectCwd` pattern through auto path — minimal change, follows established convention
- **Why not absolute paths in .barfrc**: That's a workaround in the cyrus project; the real bug is in barf-ts verification plumbing. Relative paths in `.barfrc` commands are a legitimate use case.
- **Scope**: Only fix CWD threading bugs. No refactoring of auto command structure.
- **Risk**: Low — changes are additive (new optional fields) and backward-compatible (all new fields default to undefined, preserving existing behavior when omitted)
