## Gap Analysis

**Already exists**:
- `execFileNoThrow()` correctly handles optional `cwd` parameter (passes it to `spawn()`)
- `runVerification()` in `checks.ts` accepts and passes `cwd` to `execFn`
- `runPreComplete()` accepts `cwd` in opts and passes it to `runVerification()`
- `verifyIssue()` accepts `deps?: VerifyOpts` with `cwd` and threads it through `makeVerifyPhases()` → `runChecksPhase()` → `runPreComplete()`
- `RunLoopDeps` has `projectCwd?: string` — used by `runLoop()` call sites (`outcomes.ts`, `loop.ts`)
- `barf-worker.ts` extracts `projectDir` from `WorkerRequest` and passes `cwd: projectDir` for the `verify` case
- Tests exist for `runVerification` CWD forwarding and `verifyIssue` CWD forwarding
- Python venv at `cyrus2/.venv/bin/python` exists and works (Python 3.13.5)
- Ruff lint checks pass (`ruff check cyrus2/`)

**Needs building** — Two categories:

### A. Cyrus project formatting (3 files)

`ruff format --check` fails on 3 files (trivial whitespace fixes):
1. `cyrus2/cyrus_config.py` — multi-line `float()` call should be single-line
2. `cyrus2/cyrus_server.py` — `add_argument()` kwargs need one-per-line formatting
3. `cyrus2/tests/test_027_cyrus_config.py` — tuple items need one-per-line formatting

### B. barf-ts CWD threading (4 bugs — all verified 2026-03-17)

1. **`pre-complete.ts` line 58** — Fix commands (`FIX_COMMANDS`) don't receive `cwd` even when it's available. The `cwd` is destructured on line 52 but only passed to `runVerification()` (line 75-80), not to fix command execution (line 58).
   - **Exact bug**: `execFn('sh', ['-c', step.command])` — missing third arg `cwd`

2. **`orchestration.ts` line 433** — `verifyWithSession()` calls `verifyIssue({ issueId, config, provider })` WITHOUT `deps`. The `VerifyWithSessionOpts` interface (lines 383-396) lacks a `deps` field, so there's no way to pass CWD through this wrapper.
   - **Note**: `verifyWithSession()` is exported but not currently called in production (auto.ts does inline session tracking). Still a correctness fix for the exported API — prevents future bugs if auto.ts is refactored to use it.

3. **`auto.ts` lines 76-82, 125-135, 435-439** — `autoCommand()` has three CWD gaps:
   - `AutoDeps` type (line 76-82) lacks a `projectCwd` field
   - `loopDeps` construction (lines 125-135) omits `projectCwd` — all `runLoop()` calls from auto get no CWD (this also affects `handleGatePhase` paths which spread `...loopDeps`)
   - Inline verify call at lines 435-439 passes no `deps` — CWD never reaches verification for BUILT issues

4. **`barf-worker.ts` lines 211-218** — Auto case dispatches `autoCommand({ ..., deps: { db } })` without `projectCwd`, despite having `projectDir` available from the `WorkerRequest`.

**Impact**: Any project using relative paths in `.barfrc` commands (like `cyrus2/.venv/bin/python`) fails all verification when triggered via the `auto` command path. The `runLoop` path works (via `outcomes.ts` which passes CWD) but `auto`'s standalone verify for BUILT issues and all fix commands are broken. The formatting issues in cyrus additionally cause CHECK_COMMAND to fail even when CWD is correct.

## Approach

**Two-phase fix:**

1. **Phase 1: Cyrus formatting** — Run `ruff format` on the 3 failing files. This is immediate and ensures CHECK_COMMAND will pass once CWD is fixed.

2. **Phase 2: barf-ts CWD threading** — Thread `projectCwd` through the entire auto command path, matching the pattern already established in `runLoop`/`outcomes.ts`. Also fix the pre-complete fix command CWD bug which affects all code paths.

**Why this approach**: The `projectCwd` pattern already exists and works in `runLoop`/`outcomes.ts` (see `loop.ts:432`, `outcomes.ts:292,310`). The auto command simply failed to adopt it. This is a plumbing fix, not an architectural change. The cyrus formatting is a separate, independent fix needed regardless.

## Key Decisions

- **Selected approach**: Thread existing `projectCwd` pattern through auto path — minimal change, follows established convention in `outcomes.ts`/`loop.ts`
- **Why not just absolute paths in .barfrc**: That's a workaround; relative paths in `.barfrc` are a legitimate use case. The real bug is in barf-ts verification plumbing.
- **Why fix cyrus formatting too**: Even with correct CWD, CHECK_COMMAND fails on 3 files. Both fixes are needed.
- **Why fix `verifyWithSession` even though unused**: It's an exported API — correctness fix prevents future bugs when someone refactors auto.ts to use it (plan 172 proposed exactly this)
- **Scope**: CWD threading bugs + cyrus formatting. No refactoring of auto command structure.
- **Risk**: Low — changes are additive (new optional fields) and backward-compatible (all new fields default to undefined, preserving existing behavior when omitted)
