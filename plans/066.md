## Root Cause

When barf runs the `auto` command for a non-TypeScript project (like cyrus, a Python project), the `.barfrc` configures verification commands with relative paths (e.g. `BUILD_COMMAND`). These relative paths must be resolved against the **project directory**, not the barf process's working directory.

The `cwd` parameter is correctly threaded from `runLoop` ‚Üí `handleBuildCompletion` ‚Üí `verifyIssue` ‚Üí `runChecksPhase` ‚Üí `preCheckCommand`. However, the auto command dispatcher never feeds `projectCwd` into this chain ‚Äî the value is `undefined`, so `preCheckCommand` resolves the relative path `cyrus2/.venv/bin/python` against `process.cwd()` (the barf-ts directory) where it doesn't exist, producing: `Command not found: cyrus2/.venv/bin/python (resolved to cyrus2/.venv/bin/python)`.

## Approach

Surgical threading: add `projectCwd` to the 3 missing handoff points in `auto.ts`, pass it from the worker, and fix `verifyWithSession()`. No architectural changes needed ‚Äî the infrastructure is already complete.
