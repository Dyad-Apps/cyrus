- `barf-worker.ts` `case 'auto':` passes `{ db }` only (line 203) — no `projectCwd`
- `verifyWithSession` in orchestration.ts doesn't accept or forward `cwd` (lines 365-412) — latent bug for consistency
- No test asserts that `autoCommand` forwards `projectCwd` to verification or runLoop

## Root Cause Analysis

When barf runs `auto` for a managed project (e.g. cyrus at `/home/daniel/Projects/barf/cyrus/`), verification invokes commands like `cyrus2/.venv/bin/python -m ruff check`. These are **relative paths** that only resolve when the shell cwd is the project root.

**Working path** (worker → runLoop → outcomes):
```
barf-worker.ts:108  →  runLoop(..., { projectCwd: projectDir })
loop.ts:80          →  deps.projectCwd used throughout
outcomes.ts         →  verifyIssue(..., { cwd: deps.projectCwd })  ✓
```

**Broken path** (worker → autoCommand → verify):
```
barf-worker.ts:203  →  autoCommand(..., { db })           // NO projectCwd!
auto.ts:112         →  loopDeps has no projectCwd          // child runLoops get undefined
auto.ts:428         →  _verify(id, config, provider)       // NO cwd arg!
orchestration.ts:61 →  cwd = deps?.cwd                     // undefined
checks.ts           →  execFileNoThrow(cmd, args, undefined) // runs in barf's process.cwd()
```

## Approach

Thread `projectCwd` through the `autoCommand` dependency chain, matching the existing `RunLoopDeps` pattern. This is a focused plumbing fix — no new abstractions needed.

**Why this approach**: The `RunLoopDeps` type already has `projectCwd` and the entire verification pipeline already accepts `cwd`. We just need to connect the dots in the auto command path. Four files need changes, all following established patterns.
