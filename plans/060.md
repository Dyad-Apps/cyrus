## Gap Analysis

**Already exists (working correctly)**:
- Individual plan/build commands: `barf-worker.ts` passes `projectCwd: projectDir` to `runLoop` (lines 101-119)
- `RunLoopDeps` already defines `projectCwd?: string` (loop.ts line 103)
- `loop.ts` passes `projectCwd` from deps to `handleBuildCompletion` (loop.ts line 432)
- `outcomes.ts` forwards `projectCwd` as `deps: { cwd: deps.projectCwd }` to `verifyIssue` (outcomes.ts line 310)
- `OutcomeDeps` already declares `projectCwd?: string` (outcomes.ts line 40)
- `runVerification()` in `checks.ts` passes `cwd` to `execFn` (line 185)
- `preCheckCommand()` resolves relative paths using `cwd` (checks.ts lines 96-107)
- `ExecFn` type accepts optional `cwd` third parameter (execFileNoThrow.ts lines 11-15)

**Needs building — five bugs block auto-mode cwd propagation**:

### Bug 1 — Worker doesn't pass `projectDir` to `autoCommand` (ROOT CAUSE)
`barf-worker.ts` lines 211-219: `projectDir` is available in scope (line 56) but never passed:
```typescript
case 'auto':
  await autoCommand({
    provider, batch: 1, auditGate: false, config,
    deps: { db },
    // ← projectDir available but not passed
  })
```
`AutoCommandOpts` (auto.ts lines 90-101) has no `projectCwd` field.

### Bug 2 — `auto.ts` loopDeps missing `projectCwd`
Lines 125-135: `loopDeps` doesn't include `projectCwd`, so all 5 `runLoop()` call sites (lines 181, 189, 317, 354, and audit gate phase) receive `undefined` for `projectCwd`:
```typescript
const loopDeps: RunLoopDeps = {
  agentRuntime: deps.agentRuntime,
  verifyIssue: deps.verifyIssue,
  db,
  onStateChange: ...
  // ← no projectCwd
}
```

### Bug 3 — `auto.ts` direct verify call missing `deps`
Lines 435-439: BUILT issues are verified directly without `deps: { cwd }`:
```typescript
await _verify({
  issueId: issue.id,
  config: effectiveConfig,
  provider,
  // ← no deps, so cwd is undefined
})
```

### Bug 4 — `pre-complete.ts` fix commands ignore `cwd`
Line 58: `cwd` is destructured from opts (line 52) but not passed to fix command execution:
```typescript
const result = await execFn('sh', ['-c', step.command])
// Should be: execFn('sh', ['-c', step.command], cwd)
```
Compare with line 78 where `cwd` IS correctly passed to `runVerification`.

### Bug 5 (latent) — `verifyWithSession` doesn't forward `deps`
`orchestration.ts` line 433: `verifyIssue` called without `deps`. `VerifyWithSessionOpts` (lines 383-396) has no `deps` field. Not currently called from auto-mode, but will cause same bug if used.

**Evidence**: Error `sh: 1: cyrus2/.venv/bin/python: not found` — the relative path `cyrus2/.venv/bin/python` can't resolve from barf's process cwd. The venv exists at `/home/daniel/Projects/barf/cyrus/cyrus2/.venv/bin/python`.

## Approach

Thread `projectCwd` through the auto command pipeline, matching the pattern already working for individual plan/build commands. The downstream flow (loop.ts → outcomes.ts → verifyIssue) is already correct — only the top-level plumbing is missing.

**The fix chain**:
```
Worker (projectDir) → AutoCommandOpts.projectCwd → loopDeps.projectCwd → RunLoopDeps → [existing working flow]
                                                  → _verify({ deps: { cwd } })   [direct verify call]
```

**Why this approach**: Minimal, consistent fix. Aligns auto-mode with the individual-command path that already works. All changes add an optional parameter — no behavioral changes to existing working code paths.
