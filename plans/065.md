## Root Cause Analysis

The error "Command not found: cyrus2/.venv/bin/python (resolved to cyrus2/.venv/bin/python)" proves that `cwd` is `undefined` when `preCheckCommand()` runs from the auto command path. The "(resolved to X)" text matches the `firstToken` exactly — meaning no `cwd` prefix was applied.

**Call chain breakdown (broken path)**:
```
barf-worker.ts:211  → autoCommand({ ..., deps: { db } })           ❌ no projectDir
auto.ts:125         → loopDeps = { ..., NO projectCwd }            ❌ missing from loopDeps
auto.ts:435         → _verify({ issueId, config, provider })       ❌ no deps.cwd
orchestration.ts:64 → cwd = deps?.cwd = undefined                  ❌ undefined
checks.ts:98        → resolvedPath = firstToken (no cwd prefix)    ❌ BOOM
```

**Working comparison** (direct verify command):
```
barf-worker.ts:192  → verifyIssue({ ..., deps: { cwd: projectDir } })  ✅
orchestration.ts:64 → cwd = deps?.cwd = "/home/daniel/.../cyrus"       ✅
checks.ts:98        → resolvedPath = "/home/.../cyrus/cyrus2/.venv/..."  ✅
```

## Approach

Surgical fix: thread `projectCwd` through the two missing paths in `autoCommand`. No architectural changes. The pattern is already established in the direct verify command and build completion paths — we just replicate it for the auto command path.

**Why this approach over alternatives**:
- Adding `cwd` to `Config` would change the schema and `.barfrc` parsing — overkill for a threading bug
- Using `process.cwd()` as fallback would mask the real issue and break when barf server runs from a different directory
- Threading `projectCwd` explicitly matches the existing DI pattern used throughout the codebase
