# Implementation Plan: Fix verification failure — Command not found: cyrus2/.venv/bin/python

**Issue**: [070](/home/daniel/Projects/barf/cyrus/issues/070.md)
**Created**: 2026-03-18

## Gap Analysis

**Already exists**:
- Virtual environment at `cyrus2/.venv/bin/python` — works fine when invoked from `/home/daniel/Projects/barf/cyrus/`
- `.barfrc` defines `BUILD_COMMAND`, `CHECK_COMMAND`, `TEST_COMMAND` with correct relative paths
- barf-ts verification system (`core/verification/checks.ts`) hardcodes BUILD_COMMAND + check, `TEST_COMMAND` (unit tests) as default checks
- barf-ts config only reads `TEST_COMMAND` from `.barfrc` — `BUILD_COMMAND` and `CHECK_COMMAND` are **not in the KEY_MAP** and are ignored

**Needs building**:
- A `package.json` at cyrus project root that maps BUILD_COMMAND + check, and TEST_COMMAND to the python/ruff/pytest commands

## Root Cause

The barf verification system runs BUILD_COMMAND + check, and TEST_COMMAND in the project directory. The cyrus project has no root `package.json`, so bun cannot find these scripts. The `.barfrc` `BUILD_COMMAND`/`CHECK_COMMAND` fields are defined but barf-ts doesn't use them for verification — it only uses `TEST_COMMAND` for pre-completion checks.

## Approach

Create a `package.json` at the cyrus project root with scripts that delegate to the python commands defined in `.barfrc`. This is the simplest fix that works within barf's existing verification architecture.

**Why not fix barf-ts instead?** The barf-ts codebase is in a separate project (`barf-ts/`) and modifying its verification to read `BUILD_COMMAND`/`CHECK_COMMAND` would be a larger change. Adding a `package.json` is the idiomatic solution — it's what barf expects.
